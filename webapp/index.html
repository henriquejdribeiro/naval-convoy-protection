Variables & glossary

Reference for the labels, nodes, and messages drawn on the canvas. Three groups: what you see, what moves, what is decided.

Nodes — what you see

  • A · B · C · E · F — five convoy ships acting as L1 Clique-PoA validators and proof relays. Position determines which L2 they have best signal to.
  • D — commander ship (yellow ring). Only ship authorised to issue the convoy advance command.
  • HVU — High-Value Unit. The three protected ships in the centre of the convoy. Their survival is the reason the mission runs.
  • L2-Alpha box (green) — Madara α sequencer running 5 Alpha drones. Sweeps the left frontal area.
  • L2-Bravo box (purple) — Madara β sequencer running 5 Bravo drones. Sweeps the right frontal corridor.
  • L1 PoA Verifier — Geth Clique chain (6 validators) hosting the Solidity contracts for STARK proof verification, mission registry, and commander logging.
  • EX-010 · EX-011 — mission IDs. EX-010 = left area (Alpha). EX-011 = right corridor (Bravo).

Messages — what moves

  • deploy(EX-0xx) — L1 tx by ship D registering a mission spec (area, thresholds, window) on the Solidity registry contract. Phase 1.
  • submitTelemetry(cell, x, y, t) — L2 tx from a drone after sweeping a cell. Sequenced into L2 blocks by Madara. Phase 2.
  • H_α · H_β — Poseidon hash chain over all per-cell telemetry felts. Public commitment to the swarm's full sweep. Phase 3.
  • π_α · π_β — STARK proofs produced by Stone over a Cairo verification program (one per L2). Attest the SAFE conditions held without revealing telemetry. Phase 4.
  • relay(π, public_inputs) — best-signal ship picks up the proof from L2 and submits it as an L1 tx. Ship signs envelope only; the proof's truth is checked on-chain. Phase 5.
  • verifyProof(π) — the Solidity verifier contract re-runs FRI queries on L1. On success: sets safe_outcome = true and emits a mission-verified event. Phase 6.
  • advance(MAX_SPEED) — L1 tx by ship D, gated by both EX-010 SAFE and EX-011 SAFE. Emits a convoy-advance event. Phase 7.

Outcomes — what is decided

  • SAFE — verdict for one mission. True iff coverage ≥ 95 % AND time ≤ 360 s AND no contact ≥ 0.7. Enforced inside the Cairo program, re-checked by the Solidity verifier on L1.
  • Mission-verified event — L1 event emitted when proof verification succeeds. Records mission id, which ship relayed, and at what block.
  • Convoy-advance event — L1 event broadcast to all six ships, authorising the formation to move. Only ship D, only when both missions SAFE.
  • ADVANCE (canvas state) — terminal state. Both proofs anchored on L1, command issued, convoy moves through verified-safe waters. Phase 8.

Mission brief

The convoy holds while two simultaneous reconnaissance missions complete in parallel. Each L2 produces an independent STARK proof of mission compliance.

EX-010

Left frontal area — Alpha drones

  • L2 chain: L2-Alpha (Madara α)
  • Sweep pattern: zig-zag area coverage
  • Coverage threshold: ≥ 95 %
  • Detection threshold: contacts above probability p ≥ 0.7
  • Time window: ≤ 6 minutes
  • Primary relay: ship A or F
EX-011

Right frontal area — Bravo drones

  • L2 chain: L2-Bravo (Madara β)
  • Sweep pattern: corridor (width 2a)
  • Coverage threshold: ≥ 95 %
  • Detection threshold: contacts above probability p ≥ 0.7
  • Time window: ≤ 6 minutes
  • Primary relay: ship A or B

SAFE_AREA criterion — a mission concludes with SAFE_AREA if and only if (1) total coverage ≥ threshold percentage of the assigned area, (2) total execution time ≤ window, AND (3) no contacts detected above the probability threshold. A Cairo verification program (one per L2) will encode these constraints; the STARK proof will attest all three conditions held without revealing the drone telemetry that produced them.

Transparency and privacy — plain logging is all-or-nothing: public means leaking tradecraft, secret means losing accountability. ZK‑STARKs split it — the verdict is public on L1, the telemetry that earned it stays on L2.

PUBLIC · L1
  • area
  • coverage_threshold ≥ 95 %
  • detection_threshold  p ≥ 0.7
  • time_window ≤ 360 s
  • H_α, H_β  (Poseidon commitments)
  • π_α, π_β  (proof bytes)
  • safe_outcome
  • relay_ship, block_number
PRIVATE · L2
  • cell_id, x, y, t
  • p_contact, bearing
  • waypoints, heading
  • signal_strength
  • resweep_flag
  • execution_trace  (Cairo VM)

Public = what was promised and whether it was kept. Private = how it was kept.

References

The software, specifications, and foundational papers behind every component named on this page.

Layer 1 — settlement

  • Ethereum — base protocol
  • go-ethereum (Geth) — execution client used for the local PoA chain
  • EIP-225 — Clique PoA — the consensus engine used by the six ship validators
  • Solidity — language for the verifier, registry, and command-log contracts

Layer 2 — execution & indexing

  • Starknet — STARK-based L2 the stack is modelled on
  • Madara — Substrate-based Starknet sequencer (one per drone swarm)
  • Pathfinder — Starknet full-node / indexer
  • Cairo — provable language for the SAFE-criterion program
  • The Cairo Book — official language reference

Proving pipeline

  • Stone prover — STARK prover over Cairo execution traces
  • SNOS — Starknet OS, bridges the L2 trace into the prover

Cryptographic foundations

  • ZK-STARKs — Ben-Sasson, Bentov, Horesh, Riabzev (2018) — Scalable, transparent, and post-quantum secure computational integrity
  • Poseidon — Grassi, Khovratovich, Rechberger, Roy, Schofnegger (2019) — A new hash function for zero-knowledge proof systems
  • FRI — Ben-Sasson, Bentov, Horesh, Riabzev (2017) — Fast Reed-Solomon Interactive Oracle Proofs of Proximity

Hardware-in-the-loop & network

  • ArduPilot — autopilot stack for SITL drone simulation (Phase 4)
  • ArduPilot SITL — software-in-the-loop simulator
  • tc / netem — Linux network-emulation for radio-link chaos
  • Docker — container runtime for Phases 2–4
