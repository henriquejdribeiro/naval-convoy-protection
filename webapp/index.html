Variables & glossary

Reference for the labels, nodes, and messages drawn on the canvas. Three groups: what you see, what moves, what is decided.

Nodes — what you see

  • A · B · C · E · F — five convoy ships acting as L1 Clique-PoA validators and proof relays. Position determines which L2 they have best signal to.
  • D — commander ship (yellow ring). Only ship authorised to issue the convoy advance command.
  • HVU — High-Value Unit. The three protected ships in the centre of the convoy. Their survival is the reason the mission runs.
  • L2-Alpha box (green) — Madara α sequencer running 5 Alpha drones. Sweeps the left frontal area.
  • L2-Bravo box (purple) — Madara β sequencer running 5 Bravo drones. Sweeps the right frontal corridor.
  • L1 PoA Verifier — Geth Clique chain (6 validators) holding the Solidity contracts: ConvoyProofVerifier, ConvoyMissionRegistry, ConvoyCommandLog.
  • EX-010 · EX-011 — mission IDs. EX-010 = left area (Alpha). EX-011 = right corridor (Bravo).

Messages — what moves

  • deploy(EX-0xx) — L1 tx by ship D registering a mission spec (area, thresholds, window) on ConvoyMissionRegistry. Phase 1.
  • submitTelemetry(cell, x, y, t) — L2 tx from a drone after sweeping a cell. Sequenced into L2 blocks by Madara. Phase 2.
  • H_α · H_β — Poseidon hash chain over all per-cell telemetry felts. Public commitment to the swarm's full sweep. Phase 3.
  • π_α · π_β — STARK proofs produced by Stone over the Cairo program convoy_alpha_verify / _bravo_. Attest the SAFE conditions held without revealing telemetry. Phase 4.
  • relay(π, public_inputs) — best-signal ship picks up the proof from L2 and submits it as an L1 tx. Ship signs envelope only; the proof's truth is checked on-chain. Phase 5.
  • verifyProof(π) — Solidity STARK verifier in ConvoyProofVerifier re-runs FRI queries. On success: sets safe_outcome = true, emits MissionVerified. Phase 6.
  • advance(MAX_SPEED) — L1 tx by ship D, gated by both EX-010 SAFE and EX-011 SAFE. Emits ConvoyAdvance event. Phase 7.

Outcomes — what is decided

  • SAFE — verdict for one mission. True iff coverage ≥ 95 % AND time ≤ 360 s AND no contact ≥ 0.7. Enforced in Cairo, re-checked on L1.
  • MissionVerified(id, relay, block) — L1 event when verifyProof succeeds. Records which ship relayed and at what block.
  • ConvoyAdvance(window, time, MAX_SPEED) — L1 event broadcast to all six ships, authorising the formation to move. Only ship D, only when both missions SAFE.
  • ADVANCE (canvas state) — terminal state. Both proofs anchored on L1, command issued, convoy moves through verified-safe waters. Phase 8.

Mission brief

The convoy holds while two simultaneous reconnaissance missions complete in parallel. Each L2 produces an independent STARK proof of mission compliance.

EX-010

Left frontal area — Alpha drones

  • L2 chain: L2-Alpha (Madara α)
  • Sweep pattern: zig-zag area coverage
  • Coverage threshold: ≥ 95 %
  • Detection threshold: contacts above probability p ≥ 0.7
  • Time window: ≤ 6 minutes
  • Primary relay: ship A or F
EX-011

Right frontal area — Bravo drones

  • L2 chain: L2-Bravo (Madara β)
  • Sweep pattern: corridor (width 2a)
  • Coverage threshold: ≥ 95 %
  • Detection threshold: contacts above probability p ≥ 0.7
  • Time window: ≤ 6 minutes
  • Primary relay: ship A or B

SAFE_AREA criterion

A mission concludes with SAFE_AREA if and only if:

  1. Total coverage ≥ threshold percentage of the assigned area, AND
  2. Total execution time ≤ window, AND
  3. No contacts detected above the probability threshold.

The Cairo program convoy_alpha_verify.cairo (resp. convoy_bravo_verify.cairo) encodes these constraints. The STARK proof attests that all three conditions held without revealing the drone telemetry that produced them.

Architecture at a glance

One Layer 1 settlement chain coordinates two parallel Layer 2 proving stacks. Six PoA validator ships act as proof relays; the cryptographic ground truth lives in the L1 verifier contract.

Layer 1 — Geth Clique PoA

Six validator ships (A–F) sign blocks with their PoA keys. Three Solidity contracts: ConvoyProofVerifier (deployed twice, one per L2), ConvoyMissionRegistry (per-mission status), and ConvoyCommandLog (the commander's advance commands).

Layer 2 — Madara × 2

One Madara sequencer per drone swarm. Each runs the Cairo VM, executes drone telemetry transactions, produces blocks plus an execution trace, and feeds the trace into its own SNOS → Stone proving pipeline.

Proof relay model

The orchestrator does not submit proofs to L1 directly. The closest reachable ship (best radio signal) collects the proof bytes from L2 and submits them in an L1 transaction, where the verifier contract performs the cryptographic check.

Two-of-two governance

Ship D (commander) issues the convoy advance command only after both EX-010 SAFE and EX-011 SAFE events appear on L1. A single advance command transaction is broadcast as an L1 event to all six ships.

Ship roles in the convoy
